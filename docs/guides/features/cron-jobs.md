# Cron Jobs

The **Cron Jobs** view manages scheduled commands that run while the Project Manager app is open. Use it to fire a build script every 30 minutes, refresh a remote sync every 4 hours, or run any other command-line task on a recurring interval, per project.

## At a glance

Open **Cron Jobs** from the sidebar (under the *Execution* group). The view loads the `cronJobs` list from the currently-selected project's `.project-manager/config.json` and renders a three-row stats summary, the jobs table, and the most recent run history.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ CRON JOBS                                                       [+ New Job]│
│ Scheduled commands that run while the app is open.                          │
├─────────────────────────────────────────────────────────────────────────────┤
│ ┌─────────────┐ ┌──────────────┐ ┌──────────────┐                           │
│ │ ENABLED     │ │ HISTORY      │ │ NEXT WAKE    │                           │
│ │ 3 / 5       │ │ 17 runs      │ │ in 12m       │                           │
│ └─────────────┘ └──────────────┘ └──────────────┘                           │
├─────────────────────────────────────────────────────────────────────────────┤
│ JOBS                                                                        │
│ ☑ Hourly sync                                                               │
│   git fetch --all                Every 1 hours   in 38m   ✓ 22m ago  ✎ 🗑   │
│ ☑ Smoke test                                                                │
│   npm run test:smoke             Every 30 minutes  in 12m  ✗ 28m ago ✎ 🗑   │
│ ☐ Nightly metric dump (disabled)                                            │

```

## Stats row

| Card | Value | Meaning |
|---|---|---|
| **Enabled** | `<enabled> / <total>` | How many jobs in this project are currently armed. |
| **History** | `<count> runs` | Total cron runs recorded since the last app start (capped at 100). |
| **Next Wake** | `in <Xm>` / `now` / `—` | When the soonest enabled job is due. `—` means no jobs are enabled. |

The "next wake" calculation re-renders every 10 seconds.

## The jobs table

Each row in the jobs table represents one `CronJob`:

| Column | What it shows |
|---|---|
| Toggle | A checkbox on the left. Click to enable / disable without opening the editor. |
| Name + command | Bold job name; below it, the command and args in monospace. Disabled jobs get a faint `disabled` tag next to the name. |
| Schedule | The interval — for example `Every 30 minutes` or `Every 2 hours`. |
| Next run | Countdown until the next fire — `in 12m`, `in 2h 14m`, `now`, or `—` for disabled jobs. |
| Last run | Coloured icon + relative time. Green check (`ok`), red cross (`error`), or `—` if the job has never fired. |
| Actions | Pencil (edit) and trash (delete) buttons. |

When a job is created it is scheduled `<value>` units into the future, so a freshly-added "every 30 minutes" job will fire 30 minutes after you save it — not at the top of the next half hour.

## Creating or editing a job

Click **+ New Job** (top right) or the pencil icon on a row. A right-side drawer opens with three fieldsets:

| Fieldset | Fields |
|---|---|
| **Basics** | `Name` (required), `Description`, `Enabled` toggle. |
| **Schedule** | `Every` (number, ≥ 1) + `Unit` (`minutes` / `hours`). |
| **Command** | `Command` (required), `Args` (space-separated tokens), `Working dir` (absolute path). |

The Save button stays disabled until both `Name` and `Command` are filled in. Args are split on whitespace at save time, so `--watch --silent` and `--watch   --silent` produce the same list. Working dir is passed verbatim to the underlying spawn call.

> **Schema note (PR1 of ADR-012).** The inline form today only edits `run-command` jobs. Jobs whose action type is `dispatch-engineer` — generated by the Engineers view — show up in the table and are toggleable, but the pencil button no-ops with a warning in the console. A dedicated dispatch-engineer editor lands in a follow-up PR.

## Action types

| Type | What runs | Where it's edited |
|---|---|---|
| `run-command` | Spawns a process via `spawnAgent`. Output is captured into the Logs view, just like a manual dispatch. | The form drawer in this view. |
| `dispatch-engineer` | Routes through `call_anthropic` with an engineer role's primary model and a prompt template. | (Planned) — created from the Engineers view today. |

Both types appear in the same table and share the same enable/disable toggle and run history.

## Scheduling and the heartbeat

The scheduler lives in the main client, not in the view:

```
MainClient (every 30s tick)
  └── for each enabled job:
        if Date.now() >= nextRunRef[jobId]:
          1. push nextRunRef forward by schedule.value × unit
          2. spawnAgent(command, args, workingDir)
          3. record { jobName, firedAt, status, pid } in cronHistory
          4. patch the job's lastRun / lastStatus in config.json
```

Key consequences:

| Behaviour | Why |
|---|---|
| Jobs only fire while the app is open. | There is no background process; closing the desktop shell stops the scheduler. |
| Heartbeat is fixed at 30 seconds. | So `Every 1 minute` jobs will fire on a 30-second granularity at worst. |
| `nextRunRef` is bumped **before** the spawn. | Prevents a slow command from being double-fired by a concurrent tick. |
| `cronHistory` keeps the last 100 runs in memory. | Older entries roll off the bottom; restart clears the history. |
| Job edits do not reset `nextRun`. | Toggling enable / disable preserves the existing schedule slot. |

The bridge between this view and the spawn layer is `spawnAgent` in [`lib/bridge/index.ts`](https://github.com/jason660519/Project-Manager/tree/main/lib/bridge/index.ts). All cron runs reach `lib.rs`'s `spawn_agent` Tauri command — the same path used by manual dispatches — so any environment quirks affecting one affect the other.

## Run history

Below the jobs table, the last 20 cron firings are shown in reverse chronological order:

| Field | Shown as |
|---|---|
| Status | Green check (`ok`) or red cross (`error`). |
| Name | The job name at the time it fired. |
| PID | `PID <n>` for successful spawns. Absent on errors that fail before the process is created. |
| Time | Relative — `28m ago`, `1h ago`, `2d ago`. |

For the full history grouped by day, switch to the **Logs view → Cron tab**.

## Empty states

| Situation | What you see |
|---|---|
| No project selected | The view is not rendered — pick a project on the dashboard first. |
| No jobs in this project | Dashed-border placeholder: "No jobs yet. Create one to get started." |
| No history yet | The Recent Runs section is hidden until at least one job has fired. |
| Spawn fails (e.g. command not on `$PATH`) | A row appears in history with the red cross and no PID. The job's last status flips to `error`. |

## Saved on disk

Jobs are persisted into the project config at `<projectRoot>/.project-manager/config.json` under `cronJobs[]`. The schema is part of the canonical project shape; bumping it triggers a `schemaVersion` increment (ADR-002). Disabling a job in the UI flips `enabled: false` on the JSON record but leaves the rest of the definition intact — useful when you want to pause a job over a holiday without losing the configuration.

## Follow-up integration ideas

| Follow-up | Why it matters |
|---|---|
| Cron-expression schedule | Today the only schedule shape is "every N (minutes / hours)". A standard cron expression would unlock "weekdays at 09:00", "first of the month", etc. |
| Dispatch-engineer editor | PR2 of ADR-012 — let users build dispatch-engineer jobs from this view, not just see them. |
| Background scheduling | The scheduler runs in-renderer, so closing the app stops everything. A Rust-side scheduler would survive lid-close. |
| Skip / catch-up on missed runs | After sleep / suspend, the heartbeat may miss several intervals; today they are dropped. |
| Per-job retry policy | Failed spawns currently don't retry. |
| Notification on failure | Surface red runs through the Channels view (e.g. ping Telegram). |
| Manual "Run now" button | Today you can edit the schedule to fire sooner, but a one-shot trigger would be cleaner. |
